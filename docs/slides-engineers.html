Control Flow: If/Else Conditions

Skip tasks based on runtime expressions

tasks:
  - id: check-credit
    taskRef: credit-check
    input:
      customerId: "{{input.customerId}}"

  - id: process-payment
    condition:
      if: "{{tasks.check-credit.output.approved}} == true"
    taskRef: charge-card
    dependsOn: [check-credit]

  - id: send-rejection
    condition:
      if: "{{tasks.check-credit.output.approved}} == false"
    taskRef: send-rejection-email
    dependsOn: [check-credit]

Operators: == != > < >= <= && || !

Control Flow: Switch/Case

Route to different tasks based on value

tasks:
  - id: route-payment
    switch:
      value: "{{input.paymentMethod}}"
      cases:
        - match: "stripe"
          taskRef: stripe-charge
        - match: "paypal"
          taskRef: paypal-charge
        - match: "invoice"
          taskRef: create-invoice
      default:
        taskRef: unknown-payment-error

First match wins • Falls through to default if no match

Control Flow: forEach Iteration

Iterate over arrays with parallel execution

tasks:
  - id: process-items
    forEach:
      items: "{{input.orderIds}}"      # Array to iterate
      itemVar: "order"                  # Variable name
      maxParallel: 5                    # Concurrency limit
    taskRef: process-order
    input:
      orderId: "{{forEach.order}}"
      index: "{{forEach.index}}"

output:
  results: "{{tasks.process-items.forEach.outputs}}"
  count: "{{tasks.process-items.forEach.successCount}}"

Templates: {{forEach.order}} {{forEach.index}} {{forEach.outputs}} {{forEach.successCount}}

Control Flow: Nested Patterns

Combine constructs for complex flows

# Condition inside forEach - filter during iteration
tasks:
  - id: process-orders
    forEach:
      items: "{{input.orders}}"
      itemVar: "order"
      maxParallel: 10
    condition:
      if: "{{forEach.order.status}} == 'pending'"
    taskRef: process-order
    input:
      orderId: "{{forEach.order.id}}"
Pattern Use Case
Condition inside forEach Filter items during iteration
forEach inside Condition Conditional iteration
Nested forEach Process arrays of arrays

Fail-Fast Validation

Every error caught at design time is a production incident prevented

↓ Navigate down through 5 validation layers

1. Schema Validation

JSON Schema validates all inputs and outputs at definition time

input:
  type: object
  properties:
    userId: { type: string }  # ← Enforced!
    amount: { type: integer }
  required: [userId, amount]
✗ Input "userId" expected string, got number

2. Type Compatibility

Task outputs must match downstream input types - checked before deployment

tasks:
  - id: fetch-user
    taskRef: get-user        # outputs: { email: string }
  - id: notify
    taskRef: send-email
    input:
      recipient: "{{ tasks.fetch-user.output.email }}"
✗ "fetch-user" output.email (string) incompatible with "send-sms" input.phone (integer)

3. Template Expression Validation

All template references verified - no runtime "undefined" surprises

tasks:
  - id: notify
    input:
      message: "Hello {{ tasks.fetch-user.output.emial }}"
      #                                         ↑ TYPO!
✗ Template "{{tasks.fetch-user.output.emial}}" - did you mean "email"?

4. Dependency Graph Validation

Circular dependencies and missing references caught immediately

tasks:
  - id: task-a
    dependsOn: [task-b]   # ← task-a waits for task-b
  - id: task-b
    dependsOn: [task-a]   # ← task-b waits for task-a → DEADLOCK!
✗ Circular dependency detected: task-a → task-b → task-a

5. Kubernetes Admission Webhooks

Invalid workflows rejected at kubectl apply - never reaches the cluster

$ kubectl apply -f my-workflow.yaml
Error from server: admission webhook "workflow.validate" denied the request:
  - Schema validation failed for task "process-order"
  - Circular dependency: task-a → task-b → task-a
✓ If it deploys, it works. Guaranteed.

Architecture Overview

WorkflowTask CRD

Define reusable, validated integration tasks 

apiVersion: workflow.io/v1
kind: WorkflowTask
metadata:
  name: send-notification
spec:
  http:
    method: POST
    url: https://api.notifications.internal/send
    headers:
      Authorization: "Bearer {{secrets.api_key}}"
  input:
    type: object
    properties:
      user_id: { type: string }
      message: { type: string }
      channel: { type: string }
    required: [user_id, message]
  output:
    type: object
    properties:
      notification_id: { type: string }
      status: { type: string }

Workflow CRD

Compose tasks with dependencies and data flow 

apiVersion: workflow.io/v1
kind: Workflow
metadata:
  name: user-onboarding
spec:
  input:
    type: object
    properties:
      user_id: { type: string }
  tasks:
    - id: get-user
      taskRef: fetch-user
      input: { user_id: "{{input.user_id}}" }
    - id: validate
      taskRef: validate-data
      dependsOn: [get-user]
      input:
        email: "{{tasks.get-user.output.email}}"
        profile: "{{tasks.get-user.output.profile}}"
    - id: notify
      taskRef: send-notification
      dependsOn: [validate]
